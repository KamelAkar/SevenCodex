# SevenCodex Pipeline

This document describes the current SevenCodex data pipeline at a practical level.
It covers the generator inputs, the generated JSON outputs, the EN/FR content model, asset copying, and the SevenMap integration contract.

## Purpose

SevenCodex is built as a static-first companion to SevenMap.
The generator turns FModel exports plus existing SevenMap map data into normalized JSON under `data/`, with copied assets under `assets/generated/`.

The goal is to keep the codex deployable without a backend or the extraction toolchain.

## Current generator

The pipeline currently lives in:

- `scripts/build_codex_data.py`

It uses:

- `scripts/resource_resolver.py` for localization and item/drop-group lookups
- `../SevenMap/site/data/map_data.json` and `../SevenMap/site/data/map_data_full.json` for map-linked points and region coverage
- exported game tables under `FModel/Output/Exports/SevenDeadlySins/Content/TextDatas/CData`
- exported UI and icon assets under `FModel/Output/Exports/SevenDeadlySins/Content/UI/UI_Resourse` and `FModel/Output/Exports/SevenDeadlySins/Content/UIImg`

## Rerun command

Run from the repository root:

```powershell
python scripts/build_codex_data.py
```

The script overwrites the generated JSON outputs in `data/` and refreshes copied assets under `assets/generated/`.

## EN/FR model

The codex uses a two-layer language model.

### UI language

UI chrome is localized separately in the app runtime.
That layer is not generated by this pipeline.

### Content language

Generated content records carry both English and French fields directly in the JSON.

Typical record shape:

```json
{
  "locale": {
    "en": {
      "name": "Activated Sea Grapes",
      "description": "A special grape imbued with the ocean's energy.",
      "summary": "24 tracked locations across 3 regions."
    },
    "fr": {
      "name": "Raisins de mer actifs",
      "description": "Un raisin special qui est impregne de l'energie des oceans.",
      "summary": "24 emplacements suivis sur 3 regions."
    }
  }
}
```

Rules used by the pipeline:

- English and French are both emitted when locres coverage exists.
- French fallback text is explicit when a localized string is missing.
- Slugs stay stable and technical; only labels are localized.
- Search text is flattened from both languages so queries can match either locale.

## Region model

Region pages are generated from `../SevenMap/site/data/map_data_full.json`.
The current region payloads include:

- `id`
- `slug`
- `kind`
- `locale.en`
- `locale.fr`
- `regions`
- `regionIds`
- `mapRef`
- `mapLinked`
- `searchText`

Region French labels are sourced from the local SevenMap localization vocabulary so the codex stays aligned with the map product.

## Asset copying behavior

The generator resolves icons from both the existing SevenMap asset cache and the FModel export tree.

Search roots:

- `../SevenMap/site/assets/icons`
- `../SevenMap/site/assets/subcategories`
- `../SevenMap/site/assets/game-icons`
- `FModel/Output/Exports/SevenDeadlySins/Content/UIImg`
- `FModel/Output/Exports/SevenDeadlySins/Content/UI/UI_Resourse`

Copied outputs go to:

- `assets/generated/<bucket>/...`

Behavior:

- copied assets are addressed with relative site URLs such as `./assets/generated/...`
- files are copied with timestamp-preserving `shutil.copy2`
- identical source and target paths are not recopied
- copy results are cached per source path and bucket within a single run

## Search index

`data/search-index.json` is a prebuilt search payload.
It is generated from the normalized codex entries rather than being inferred in the browser from raw tables.

The current search index contains:

- localized names
- descriptions and summaries
- list kind and subkind
- region labels
- flattened search text for accent-insensitive matching

## SevenMap integration contract

The intended contract is one-way and two-way:

- SevenCodex detail pages should link to SevenMap when an entry is mappable.
- SevenMap detail panels should link back to the corresponding SevenCodex entry when one exists.

Practical deep-link model:

- `?region=C01-LIONES`
- `?type=gathering&subcategory=kudzu-vine`
- `?type=pet&petItemId=123456789`
- `?pointId=deff0834d9a15f09`

Back-link metadata on the map side should expose:

- `codexId`
- `codexKind`
- `codexSlug`

The map contract is intentionally simple so the codex can stay static and the map can keep its existing runtime shape.

## Practical notes

- The pipeline is reproducible from local exports and the current SevenMap map data.
- Generated outputs are meant to be versioned as build artifacts, not hand-edited.
- If a new game table is added later, extend the generator and document the new source table here.
- If a new codex category is added later, update the manifest/list contract and keep the search index in sync.
